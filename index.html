Básico — Módulo 0

¿Por qué existe BGP?

El Internet no es una red única — es una red de redes. BGP es el protocolo que las une.

🌍 Caso real En 2008, Pakistan Telecom anunció por error una ruta BGP que decía ser el camino más corto hacia YouTube. En minutos, el tráfico mundial de YouTube se redirigió hacia Pakistán y el servicio cayó globalmente por 2 horas. Eso es el poder (y el riesgo) de BGP.
El problema que BGP resuelve

🏢 El Internet como grafo

Miles de entidades administrativas independientes — llamadas Autonomous Systems — cada una con su propia red interna. El problema: ¿cómo sabe una red cómo llegar a otra que está en un AS distinto?

La respuesta es BGP: cada AS es un supernode, y las sesiones BGP son los virtual links que forman el grafo del Internet.

🔀 Por qué no usar OSPF para todo

Link-state (OSPF): cada router conoce la topología completa. No escala con millones de routers en el Internet.

Distance vector (RIP): escala mejor, pero crea loops. No es confiable para el Internet.

BGP — Path Vector: evita loops con el AS-PATH, usa TCP para entrega confiable, y escala mediante jerarquía de ASes.

La jerarquía del Internet
Tier 1

Backbone Global

No pagan tránsito a nadie. Peering gratuito entre ellos. Ejemplos: AT&T, NTT, Level3.

Tier 2

ISP Regional

Compran tránsito a Tier 1. Hacen peering con otros Tier 2. Claro (AS 11664) opera en este nivel en Latinoamérica.

Tier 3 / Cliente

Usuario Final

Compran tránsito a su ISP. Pueden tener ASN privado. Ejemplo: AS 64518 del cliente en tu proyecto.

Básico — Módulo 1

Terminología fundamental

Los bloques conceptuales que necesitás antes de entender cómo opera BGP.

Core

Autonomous System (AS)

Colección de redes IP y routers bajo una misma política administrativa de routing. Administrado por una sola organización.

Identificado por un ASN. Cada AS decide sus propias políticas: qué rutas acepta, qué anuncia, a quién le da tránsito.

Números

ASN — Autonomous System Number

Públicos: 1–64495 y 131072–4294967295. Asignados por LACNIC/ARIN. Se propagan en internet.

Privados: 64512–65534. Solo para uso interno — nunca deben llegar a internet. Tu caso: AS 64518 (privado del cliente).

Core

Prefijo IP / Network

En BGP, una network es un bloque de IPs en notación CIDR. Ejemplo: 200.81.251.0/24 (256 IPs).

BGP anuncia la alcanzabilidad de estos prefijos. No maneja IPs individuales — trabaja con bloques.

Roles

BGP Speaker y Peer

Speaker: router que ejecuta BGP. Es el punto de entrada/salida de anuncios para su AS.

Peer / Neighbor: dos speakers con sesión BGP entre sí. La relación se configura manualmente — BGP no descubre peers automáticamente.

Bloques PI vs PA — importante para tu proyecto

Bloque PI — Provider Independent

Asignado directamente al cliente por LACNIC/ARIN. El cliente lo posee. Puede anunciarlo desde cualquier proveedor.

✓ Permite el esquema que el cliente quiere: mismo prefijo anunciado desde ambos sitios.

Bloque PA — Provider Aggregatable

Asignado por el ISP al cliente. Pertenece al ISP. Si el cliente cambia de proveedor, pierde el bloque.

⚠ El prefijo 200.81.251.0/24 provisto por Claro es PA — solo Claro puede anunciarlo.

Básico — Módulo 2

Sesión BGP y mensajes

Cómo dos routers establecen y mantienen una sesión BGP, y qué mensajes intercambian.

Los 4 mensajes BGP
Tipo 1

OPEN

Inicia la sesión. Contiene: versión BGP (4), ASN del sender, Hold Time propuesto, BGP Identifier (IP del router, generalmente loopback).

Tipo 2

UPDATE

Anuncia nuevas rutas y/o retira rutas existentes en el mismo mensaje. Contiene: path attributes + prefijos NLRI (Network Layer Reachability Info).

Tipo 4

KEEPALIVE

Mantiene viva la sesión. Se envía cada 1/3 del Hold Timer (~30s). Si no se recibe antes de que expire el Hold Timer (90s), la sesión cae.

Tipo 3

NOTIFICATION

Reporta un error y cierra la sesión inmediatamente. Contiene código de error y sub-código. Después de enviar NOTIFICATION, el TCP se cierra.

Máquina de estados FSM — click en cada estado
Timers críticos

Hold Timer

Tiempo máximo sin mensajes antes de declarar la sesión caída.
Default: 90 segundos.

Keepalive Timer

Frecuencia de envío de KEEPALIVE = 1/3 del Hold Timer.
Default: 30 segundos.

Connect Retry Timer

Tiempo de espera antes de reintentar conexión TCP cuando falla.
Usado en estados Connect y Active.

Básico — Módulo 3

eBGP vs iBGP

Las dos caras de BGP: entre ASes diferentes, y dentro del mismo AS.

External BGP

eBGP — Entre ASes distintos

Sesión entre routers frontera de dos ASes diferentes. Es la forma clásica de BGP.

Comportamiento clave: cada vez que un UPDATE cruza un límite de AS, el número del AS saliente se prepend al AS-PATH. Esto es el mecanismo anti-loop.

Ejemplo: AS 64518 ↔ AS 11664 (Claro). Cada vez que el UPDATE cruza esa frontera, el ASN se agrega.

Internal BGP

iBGP — Dentro del mismo AS

Sesión entre routers dentro del mismo AS. Necesario cuando un AS tiene múltiples BGP speakers conectados a otros ASes.

El AS number no se añade al AS-PATH en sesiones iBGP. Esto crea el problema de loops internos → origen de las 2 Reglas.

Las 2 Reglas críticas del iBGP

Regla 1 ✓

Un BGP speaker puede anunciar a un iBGP neighbor los prefijos aprendidos de un eBGP speaker, y viceversa.

Esto es lo que permite que las rutas externas se distribuyan dentro del AS.

Regla 2 ✗

Un iBGP speaker NO puede anunciar a otro iBGP speaker los prefijos que aprendió de un iBGP speaker.

Sin esta regla, los loops son posibles dentro del AS porque el ASN no se modifica en iBGP.

El problema del full-mesh

La Regla 2 implica que si hay más de 2 iBGP speakers en un AS, se necesita sesión entre cada par — full-mesh.

N×(N-1)/2
sesiones requeridas
5 routers → 10 sesiones 10 routers → 45 sesiones 50 routers → 1,225 sesiones 100 routers → 4,950 sesiones

Solución: Route Reflectors — ver Módulo 6.

Well-known Mandatory

AS-PATH

Controla loops + preferencia de ruta

Secuencia de ASNs que la ruta ha atravesado. Cada vez que un UPDATE cruza un límite de AS, el ASN del AS saliente se prepend al inicio. BGP prefiere el path más corto.

Well-known Mandatory

NEXT-HOP

IP del router que conoce el camino al destino

Identifica la IP del router frontera hacia el prefijo destino. En eBGP directo = IP del router que anuncia. En iBGP, el NEXT-HOP no cambia cuando el UPDATE se propaga internamente — sigue siendo la IP del speaker eBGP original.

En tu proyecto: 10.169.209.9 es el NEXT-HOP provisto por Claro (IP del peer BGP de Claro).

Well-known Discretionary

LOCAL-PREF

Controla la SALIDA del tráfico del AS

Métrica usada solo dentro de un AS. Mayor LOCAL-PREF = enlace preferido para tráfico de salida. No sale del AS.

LOCAL-PREF enlace principal: 200
Optional Non-Transitive

MED

Multi-Exit Discriminator — sugiere cómo ENTRAR al AS

Cuando hay múltiples enlaces hacia el mismo AS vecino, el vecino prefiere el enlace con menor MED. Solo aplica para comparar rutas del mismo AS vecino.

No se propaga más allá del AS vecino (non-transitive). El ISP puede borrar el MED antes de propagarlo.

En tu proyecto: Claro podría usar MED para indicar al cliente por cuál PoP prefiere que entre el tráfico.

Well-known Mandatory

ORIGIN

Cómo llegó el prefijo a BGP

Indica la fuente del prefijo:

  • i — IGP: introducido vía network command (más confiable)
  • e — EGP: aprendido de protocolo EGP legacy
  • ? — Incomplete: redistribuido desde otra fuente (menos confiable)

Orden de preferencia: IGP > EGP > Incomplete.

Optional Transitive

COMMUNITY

Etiquetas para políticas de routing en grupo

Valor de 32 bits que permite etiquetar rutas. Los peers que entienden el tag aplican políticas correspondientes. Es transitive: speakers que no lo reconocen lo pasan igual.

Well-known communities:

  • NO_EXPORT — no anunciar a peers eBGP
  • NO_ADVERTISE — no anunciar a nadie
  • NO_EXPORT_SUBCONFED — no exportar fuera de la confederación

Route Reflectors (RR)

El problema

La Regla 2 de iBGP exige full-mesh. Con 100 routers = 4,950 sesiones. Inmanejable para un ISP grande.

La solución: Route Reflector

Un router designado (RR) actúa como hub. Los demás son clients y solo hablan con el RR.

Reglas del RR:

  • Ruta de eBGP peer → refleja a todos los clients y non-clients
  • Ruta de un client → refleja a todos los otros clients y non-clients
  • Ruta de un non-client iBGP → refleja solo a los clients
Confederations

Alternativa al RR. El AS se divide en sub-ASes (confederation members). Cada sub-AS tiene full-mesh interno. Entre sub-ASes se usa eBGP modificado con ASNs privados. Externamente, el AS se presenta como una sola entidad.

Ventaja: más control de políticas por sub-AS. Desventaja: mayor complejidad de configuración.

Avanzado — Módulo 7

Route Flap Dampening

Cuando una ruta aparece y desaparece repetidamente, BGP penaliza esa inestabilidad para proteger el Internet.

💡 El problema Si un prefijo flappea cada 2 minutos en el core de internet (196,000 prefijos, 1% inestables), la cascada de UPDATE messages sobrecarga la CPU de cada router en el camino. Route Flap Dampening es el mecanismo de protección.
Parámetros del mecanismo

Penalty per flap

Penalidad agregada por cada flap. Default: 1,000 puntos.

Suppress Limit

Si la penalidad supera este umbral, la ruta se suprime. Default: 2,000.

Reuse Limit

La ruta se reactiva cuando la penalidad cae por debajo de este valor. Default: 750.

Half-life (H)

Tiempo en que la penalidad decae a la mitad. Default: 7.5 minutos.

Curva de penalidad — 5 flaps (cada 2 min)

Los flaps ocurren en t=2,4,6,8,10 min. La ruta se suprime cuando la penalidad cruza 2,000 (línea roja). Se reactiva cuando cae por debajo de 750 (línea verde) — aproximadamente a los 27 minutos.
